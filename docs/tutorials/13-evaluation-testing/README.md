# 第 13 章: Agent 评估与测试策略

> **核心问题**：Agent 的行为不是确定性的——同一个 Prompt 每次输出可能不同。怎么客观衡量它好不好？

## 13.1 Agent 的核心评估指标

传统软件测试（输入-预期输出-断言）在 Agent 场景下远远不够。需要多维度评估。

### 五大核心指标

| 指标 | 定义 | 衡量方法 |
|------|------|---------|
| 完成率 | Agent 成功完成任务的比例 | 端到端自动化测试通过率 |
| 准确率 | 输出内容的语义正确性 | LLM-as-Judge 打分 |
| 幻觉率 | 输出中出现事实性错误的频率 | 知识库对照 + 人工抽样 |
| 轮次 | 完成任务所需的交互次数 | 自动记录每次 Tool Call |
| 耗时 | 从输入到最终输出的总时间 | 端到端计时 |

### 指标权重矩阵

不同类型 Agent 各指标权重不同：

客服 Agent: 完成率(40%) > 准确率(30%) > 轮次(15%) > 耗时(10%) > 幻觉率(5%)
代码助手 Agent: 准确率(40%) > 幻觉率(30%) > 完成率(20%) > 耗时(10%)
数据分析 Agent: 准确率(35%) > 完成率(30%) > 耗时(20%) > 幻觉率(15%)

## 13.2 三种评估方法

### 方法一：自动化评测

自动化测试是最基础、最容易嵌入 CI/CD 的评估方式。项目中的测试文件就是典型例子。

### 方法二：LLM-as-Judge

用另一个更强的 LLM 来评估 Agent 的输出质量。Judge LLM 根据评分标准给出分数和原因。

### 方法三：红蓝对抗

红队 Agent 生成恶意/边界输入，蓝队 Agent 尝试正确应对。适合评估安全性和鲁棒性。

## 13.3 回归测试

LLM 模型升级后，同样代码可能产生不同行为。回归测试捕获这种模型漂移。

1. 记录基线：用当前模型跑 N 个测试用例，保存输出
2. 升级后重跑：用新模型跑同样用例
3. 对比差异：输出变化是否可接受？

## 13.4 幻觉衡量与治理

### 三层治理策略

从轻到重，逐层防幻觉：

1. Prompt 层：明确不知道就说不知道，不要编造
2. 工具层：强制回答事实性问题时先调用 RAG 搜索
3. 校验层：输出后用另一个 LLM 做事实校验

## 面试高频问法

Q1: 怎么衡量一个 Agent 的效果好坏？
从五个维度衡量：完成率、准确率、幻觉率、轮次、耗时。不同场景权重不同。

Q2: 怎么做 Agent 的回归测试？
核心是基线对比：升级前保存基线，升级后重跑对比。

Q3: 怎么降低 Agent 的幻觉？
三层递进：Prompt 层约束、工具层强制核查、校验层后置过滤。组合使用效果最好。
